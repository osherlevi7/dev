POC | GKE Secured Cluster

Architecture

Part 1: Building the infrastructure for the environment in Terraform is an efficient way to ensure consistency and repeatability across environments. Utilizing Terraform to deploy network infrastructure components such as Cloud Firewall, Kubernetes Engine, Secrets Manager, Cloud IAMs, Cloud Routes, Cloud Router, and VPC, creates a solid foundation for your Private GKE Cluster. Building two node pools for infra and services purposes is also a recommended best practice for managing resources effectively.

Part 2: Building the platform for the environment in Terraform provides an automated and streamlined approach for provisioning Cloud DNS, External IP for Load Balancing, Ingress Controller, Cloud Armor, Cloud SQL, Pub/Sub Messaging, and Cloud Scheduler. Deploying these dependencies as part of the platform setup ensures that the environment is configured for scalability and reliability. Utilizing Helm charts and a Helm repository can also streamline the deployment of your application and provide easy versioning and rollback capabilities.

Part 3: Creating the microservices application for the environment is an essential part of the architecture. Using Helm charts to release the local application and a Helm repository to manage the deployment of your application, provides a scalable and efficient approach to managing your application deployment. It's recommended to containerize your application for portability and ease of deployment. Additionally, incorporating automated testing and monitoring in your pipeline can help ensure the reliability and resilience of your application.

So, implementing these best practices will provide a solid foundation for building a Private GKE Cluster architecture that is scalable, reliable, and secure.


Key Components

The project is built using Terraform to create an environment infrastructure, platform, and microservices for an application.
Part 1 involves creating the network infrastructure and Kubernetes engine with two node pools for infrastructure and services. Other services, such as cloud firewall, secrets manager, IAM, and routes, are also included.
Part 2 involves creating environment dependencies, including DNS, an external IP for load balancing, ingress controller, Cloud Armor for security, Cloud SQL for database management, Pub/Sub messaging for communication, and Cloud Scheduler for scheduling tasks.
Part 3 involves creating specific application microservices using Helm charts for deployment, and includes a Helm repo that will be installed on the application itself.
The project appears to prioritize security by including services like Cloud Armor, and the infrastructure is built with network security in mind, such as using VPCs and routing tables.
The project is built using best practices for cloud infrastructure, including infrastructure as code, scalable and modular architecture, and the use of managed services to reduce maintenance and increase reliability.


  • A Private Cluster in GKE is a deployment option that isolates the master endpoint of the cluster from the public internet.
  • In a Private Cluster, the Kubernetes API server is only accessible from within a designated Virtual Private Cloud (VPC) network.
  • To access a Private Cluster, a bastion host is used as a proxy to establish a secure connection to the Kubernetes API server.
  • GKE's Node Auto-Upgrade and Kubernetes Role-Based Access Control (RBAC) features can be used to enhance the security of a Private Cluster environment.
  • Private Clusters are a more secure deployment option for organizations that require additional control and isolation of their Kubernetes clusters.
